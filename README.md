# TALIS PoC

Last updated: 2026-05-10

TALIS is a Territorial Agricultural & Land Intelligence System proof of concept for public-sector investment planning teams. This frontend demonstrates the six-page decision workflow: discovering candidate territories, inspecting territory fundamentals, reviewing intelligence signals, assessing opportunity and risk, comparing investment decisions, and monitoring policy accountability. The PoC is fixture-backed, desktop-first, and intentionally scoped to frontend workflows plus local adapter contracts.

## Live Demo

The TALIS PoC website is available at https://talis-poc.vercel.app.

## Quick Start

Prerequisites:

- Node.js 20+
- npm 10+
- Unix-like shell; use WSL on Windows

Common commands:

```bash
npm run build
npm start
npm run check
npm test
npx playwright test
ANALYZE=true npm run build
```

`npm run check` runs lint, typecheck, and Vitest. `ANALYZE=true npm run build` emits Next.js bundle analyzer reports when the analyzer dependency is installed.

## Fixture Guide

Fixtures live under `src/mocks/` and are consumed through adapters only:

- Discovery and shared territory lists: `src/mocks/dim_wilayah.json`, `src/mocks/wilayah_profil_sample.json`, `src/mocks/wilayah_score_aggregate.json`
- Territory profile: `src/mocks/territory/`
- Territory intelligence: `src/mocks/intelligence/`
- Opportunity and risk assessment: `src/mocks/assessment/`
- Investment decision: `src/mocks/decision/`
- Monitoring and governance: `src/mocks/monitoring/`

Validation happens with Zod schemas in `src/lib/schema/*`. UI components must not import from `src/mocks/` directly; component data access goes through `src/lib/api/apiClient.ts` and the adapter files under `src/lib/api/*`.

Invented or derived values are documented in:

- `src/mocks/_assumptions.md`
- `src/mocks/intelligence/_assumptions.md`

SPRINT-07 monthly monitoring snapshots are generated by `scripts/generate_snapshots.ts`.

## Dev Toggles

- Page 5 dev-only `Simulate Δ`: mutates shortlist current values for delta-tracking demonstration.
- Global dev-only `Simulate New Alert`: pushes a fake alert through the realtime stub.
- Header dev-only LCP overlay: set `NEXT_PUBLIC_TALIS_LCP_OVERLAY=1` while running `next dev` to show current page LCP.

These are development affordances and are not active in production builds.

## State Boundaries

| Store / hook | Storage | Scope |
| --- | --- | --- |
| `useActiveProfile()` | session profile state + URL query support | Global active sample profile |
| `useDiscovery()` | `sessionStorage.talis.discovery.v1` | Command Center search and Quick Scan panel state |
| `useAssessment()` | `sessionStorage.talis.assessment.session` | Page 4 active sektor; custom weights are not persisted |
| `useComparison()` | `sessionStorage.talis.comparison.v1` | Page 5 active comparison candidates |
| `useShortlist()` | `localStorage.talis.shortlist.v1` | Saved shortlist items and local snapshots |
| `useMonitoring()` | `sessionStorage.talis.monitoring.session` | Page 6 alert filters and policy period picker |

Additional local keys:

- `localStorage.talis.policy_log.v1`: local policy notes added during the PoC.
- `localStorage.talis.audit_log.v1`: SPRINT-08 export audit stub, capped at the last 100 entries.

## Realtime Stub

`src/lib/realtime/` provides an in-process EventEmitter behind a SocketIO-shaped contract. The post-PoC swap point is the `RealtimeAdapter` interface: replace the local emitter with `socket.io-client` wired to the Kafka-backed event stream.

## Export Engine

`src/lib/export/` builds client-side report Blobs:

- PDF via `@react-pdf/renderer`
- PPT via `pptxgenjs`
- XLSX via `xlsx`

Export UI lives in `src/components/export/`. Heavy format modules are loaded lazily when the export modal is opened.

## Audit Log Stub

`src/lib/audit/` writes `{ user_id, action, payload, timestamp }` to the console and `localStorage.talis.audit_log.v1`. This is a PoC simplification. The production audit trail described in `../docs/00_OVERVIEW.md` is a backend responsibility.

## Swapping To A Real Backend

1. Replace fixture-loader bodies in `src/lib/api/*` with HTTP fetch calls while keeping adapter signatures stable.
2. Keep Zod validators at the adapter boundary, but validate real HTTP responses instead of local fixtures.
3. Replace `src/lib/realtime/` with a `socket.io-client` implementation of `RealtimeAdapter`.
4. Move local persistence for shortlist, policy notes, monitoring filters, and assessment session data behind backend-backed APIs where required.
5. Replace the audit stub with a backend audit-log endpoint.

Production deployment should follow standard Next.js deployment practice, such as Vercel or a Node-hosted `next start` environment.

## PoC Scope

- Desktop-first; tablet and mobile responsive polish is not complete.
- No authentication or authorization.
- No real backend, BKPM, OSS, Airflow, or Kafka integration.
- Dev toggles are for local development only.
- Alert lifecycle persistence is a local stub.
- Audit logging is a local stub.
- Exported PDF reports do not embed interactive map imagery; templates use text/table summaries and footnotes.
- Fixture values are representative planning data, not production investment advice.
